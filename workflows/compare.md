# Workflow: Compare

> Competitive analysis and benchmarking. Phase 5 of the Human Operating System.

## When to Use

- Before claiming a feature is "good enough"
- When building something that has existing alternatives
- When the user asks "how does this compare?"
- Before launching or shipping

## Process

### Step 1: Identify the Subject and Competitors

What are we comparing?
- A feature → compare to the same feature in competing products
- A product → compare to direct competitors
- An implementation → compare to best-practice patterns
- A design → compare to modern standards

Identify 2-3 real alternatives to compare against. Use web search if needed.

### Step 2: Run Category 11 — Competitive Benchmarking

For each competitor/alternative:

| Dimension | Our Version | Competitor A | Competitor B |
|-----------|-------------|-------------|-------------|
| Core functionality | [assessment] | [assessment] | [assessment] |
| User experience | [assessment] | [assessment] | [assessment] |
| Performance | [assessment] | [assessment] | [assessment] |
| Design quality | [assessment] | [assessment] | [assessment] |
| Edge case handling | [assessment] | [assessment] | [assessment] |

For each dimension, note:
- What do they do better than us?
- What do we do better than them?
- What's table stakes that everyone must have?

### Step 3: Run Category 12 — Trust and Credibility

- Does our version look as professional as alternatives?
- Are there inconsistencies that reduce credibility?
- Would a new user trust this on first impression?
- Does it meet the quality bar set by the competition?

### Step 4: Gap Analysis

Identify:
1. **Must-fix gaps**: We're clearly worse in ways that matter
2. **Nice-to-have gaps**: They're better but it's not critical
3. **Our advantages**: Things we do better (don't lose these)
4. **Table stakes missing**: Things every competitor has that we don't

### Step 5: Report

```
## Competitive Analysis — [Subject]

### Competitors Analyzed
1. [name] — [why chosen as comparator]
2. [name] — [why chosen as comparator]

### Comparison Matrix
[table from Step 2]

### Key Gaps
- [gap] — Severity: Must-fix/Nice-to-have — [what to do]

### Our Advantages
- [advantage] — [how to protect/enhance it]

### Trust Check
[would a user choose us over alternatives? why/why not?]

### Verdict
[Competitive / Needs work in specific areas / Not competitive yet]
```

## Key Principle

Honest comparison is more valuable than flattering self-assessment.
If we're worse than alternatives in important ways, that's critical information — not a failure to acknowledge.
